Quickstart

FounderFinder in 5 minutes

  1. Open Settings and confirm Exa + Parallel are configured.
  2. Run Founder Finder and review Tier 1 and Tier 2 candidates.
  3. Import top candidates to Talent Pool.
  4. Create or update a venture in Concepts and use Matching.
  5. Track outreach in Engagements and progress in Residencies/Investment.
Operator rule: score evidence first, then make outreach decisions.

Walkthrough

End-to-end operating flow

01 | setup

Configure environment

In Settings, configure Exa, Parallel, and Harmonic. Without Exa, founder search is blocked.

02 | source

Run Founder Finder

Trigger search and enrichment. The pipeline runs Exa sourcing, EEA scoring, then Parallel enrichment.

03 | triage

Triage by score, not intuition

Prioritize Tier 1 for immediate outreach, Tier 2 for shortlist building, Tier 3 for later revisit.

04 | import

Import to Talent Pool

Import top candidates. Use batch import for Tier 1 when running weekly sourcing sprints.

05 | match

Map founders to concepts

Use Concepts + Matching to link founder profiles to active venture opportunities.

06 | engage

Run outreach and progression

Move conversations through Engagements, then track active candidates in Residencies and Investment.

07 | build

Launch with Build OS

For selected ideas, use Build OS to move from exploration to shipped product via the fixed Eli workflow.

Best Practices

How teams get the most value

Run a weekly sourcing cadence

  • One sourcing run per week with a consistent query family.
  • Batch import Tier 1 immediately.
  • Review Tier 2 in a scheduled operator pass.

Use EEA as the decision spine

  • Treat EEA tier as the first-pass gate.
  • Use false-positive flags to avoid over-scoring.
  • Add manual conviction notes only after evidence review.

Keep profile hygiene high

  • Prefer real LinkedIn/GitHub URLs over manual guesses.
  • Normalize duplicate records early.
  • Capture source channel on every import.

Close the loop in Build OS

  • Do not advance stages until artifact gates pass.
  • Always record QA notes before manual polish.
  • End every project as shipped or parked.

Concepts + Matching

  • Keep concept problem statements current before matching.
  • Link high-priority candidates to concepts within 24 hours.
  • Store explicit fit rationale in notes.

Engagements + Residencies + Investment

  • Track outreach touchpoints and next follow-up date.
  • Promote only candidates with evidence + concept alignment.
  • Capture decision notes: thesis, risks, next action.

Build OS

  • Do not skip artifact or checklist gates.
  • Keep PRD and TDD updated as project reality changes.
  • Close every project as shipped or parked.

Lever Automation

Guidelines to auto-identify strong new + previous applicants

Use Lever exports as a repeatable intake feed, then route records into FounderFinder for scoring, dedupe, and follow-up.

1) Intake and normalization

  • Export Lever candidates_by_origin weekly (or daily for high volume).
  • Keep core fields: name, email, role, company, location, stage, source, links, resume text.
  • Normalize LinkedIn/GitHub URLs before import to reduce duplicate profiles.
  • Tag each record with import date and job posting for cohort analysis.

2) Automated quality filters

  • Auto-pass if candidate has 2+ strong technical signals (publications, OSS traction, elite engineering roles).
  • Auto-pass if profile clearly shows founder/operator + technical build evidence.
  • Auto-hold if profile is incomplete or has no verifiable technical indicators.
  • Auto-reject low-fit roles when candidate objective does not match venture-stage needs.

3) Previous applicant resurfacing rules

  • Re-open previously rejected applicants after 6-12 months if new technical signals appear.
  • Prioritize prior applicants with role progression (e.g., Sr Eng → Staff/Principal).
  • Prioritize prior applicants with new founder activity, launches, or notable OSS growth.
  • Suppress resurfacing if prior rejection reason was hard disqualifier (not skill-gap based).

4) Scoring and routing formula

lever_signal_score = technical_signals + founder_signals + recency_bonus - false_positive_penalty

if score >= 80: route = "Priority Outreach"
if score 60-79: route = "Operator Review"
if score 40-59: route = "Nurture / Recheck"
if score < 40: route = "Archive"
Current best path in FounderFinder: use ATS import preview, keep isPreferredLeverExport true where possible, then run EEA-style review for final decisioning.

Recommended weekly operating loop

  1. Import new Lever export and sync candidates.
  2. Run duplicate merge pass by email + normalized LinkedIn/GitHub.
  3. Review all Priority Outreach and top Operator Review records.
  4. Trigger outreach sequences and update engagement states.
  5. Resurface previous applicants that crossed new signal thresholds.

Troubleshooting

Common issues and fast fixes

Error: Failed to send a request to the Edge Function

  1. Confirm you are logged into FounderFinder (not SourceKit).
  2. Confirm Exa is configured in Settings.
  3. Hard refresh and rerun Find Founders.
  4. If still failing, verify deployed function slugs include founder-source and founder-enrich.

No password available

  1. Use password reset from FounderFinder login page.
  2. Reset the account in FounderFinder Supabase project for the same email.
  3. Sign in and verify membership access check passes.

GitHub CLI run watch errors

Do not include angle brackets in run ids.

gh run watch 23031335682
gh run view 23031335682 --log-failed

Merge blocked by repo rules

  • Use PR flow only (no direct pushes to main).
  • Wait for required build status check.
  • Resolve all PR conversation threads before merge.
Onboarding

New user setup checklist

  1. Login works and FounderFinder app membership is active.
  2. Exa and Parallel are configured in Settings.
  3. Run one successful Founder Finder sourcing pass.
  4. Import one candidate into Talent Pool.
  5. Link one candidate to a concept in Matching.
  6. Create one engagement record and set next action date.
  7. Create one Build OS project and populate required artifacts.
Target onboarding completion time: 20 minutes for an operator, 10 minutes for a returning user.

FAQ

Common questions

  • Do I need all integrations configured? Exa is required for founder search; Parallel and Harmonic add depth.
  • How often should we run sourcing? Weekly by default; high-volume teams can run two cycles per week.
  • What should trigger outreach? Tier 1 always; Tier 2 when concept fit is clear or conviction notes are strong.
  • Can Build OS run without a concept record? Yes, concept linking is optional.
  • What is the strongest operator habit? Keep artifact quality high and close every loop with explicit stage outcomes.
